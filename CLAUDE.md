## Solver architecture

`src/solver.cpp` (single file, C++20, ~1000 lines) — multithreaded (default cores−2), default `--mode mix`.

**Hunter** (full-board constructor):
- Randomized *perfect* border-ring DFS, then row-major interior DFS under a mismatch budget: exact candidate buckets `[west][north]` + mismatch lists, color-deficit lower bound, perfect-prefix rows with `pre_budget`.
- Anytime B&B: each completion tightens budget to `cost−1`; self-tightening global budget. Also `greedy_complete()` (min-mismatch per cell, ~390–405 score) for fast pool bootstrap.

**Polisher (LNS)** — takes a completed board and repeatedly:
- Re-solves regions *exactly* via `RegionSolver` B&B: 50% rectangles (sides 2–5) / 50% fault-blobs, up to `region_max=16` cells, `region_node_cap=2M`.
- **Incumbent seeding (crucial)**: B&B starts with the original arrangement as initial best (`best_gain = inc_gain − 1`), so capped searches can never lose ground, and pruning is ~1000× stronger (exhaustive to n≈16). Ties may replace incumbent → deliberate sideways drift (per-node candidates sorted by immediate gain desc, random tie-break; per-candidate break bound `gain + cand.g + suffix_max[i+1] <= best_gain`).
- Hungarian O(n³) optimal reassignment of pairwise non-adjacent fault cells ("scatter").

**Two-tier seed pools** (fixes lineage trapping): `g_seeds` elite (score ≥ best − seed_slack) + `g_fresh` (all completions). `ensure_board`: 30% fresh / else elite / else fresh / else global best. **Gotcha fixed once already**: mix_thread's bootstrap gate must check `g_fresh.size() < 8`, not `g_seeds` — gating on the elite pool deadlocks polish forever when resuming a high best (greedy completions can't reach the elite threshold).

Other facts:
- Clue cells pinned, rim always grey-correct → every reported score is record-eligible. `board_conforms()` enforces this on resume/seed.
- `--seed-edges` accepts a full bucas URL or bare 1024-letter string.
- `--minutes N` watchdog for timed runs; status line every 15 s (best/restarts/avgdepth/nodes/polish/seeds/fresh).
- Validation: `tools/e2lib.py` independently reproduces C++ scores exactly (piece multiset, clue placement, grey border all cross-checked).
- Build: `scripts\build.ps1` (g++ full WinGet WinLibs path, `-O3 -march=native -funroll-loops -std=c++20 -static -pthread`).

Tuning ideas not yet done: polish_frac tuning; full per-node Hungarian bound in RegionSolver (only the top-2 dynamic suffix bound is in); GPU region B&B.

**2026-07-10 Verhaard adoptions** (from analyzing https://www.shortestpath.se/eii/eii_details.html — Louis Verhaard's 2008 prize-winning 467 solver writeup; all in `src/solver.cpp`, all opt-in flags except the mix-thread fix):
- **Graded slip schedule (`--slip-target N`, THE from-scratch breakthrough)**: replaces the `free_row`/`pre_budget` step function with LV's cumulative per-depth mismatch caps — perfect prefix for the first ~70% of the scan, then caps rise 1 at a time with shrinking gaps (sqrt spacing) across the last `--slip-span` (0.30) of the scan, topping out at `480−N`; the last ~span/10 cells get no new slips (LV holds the final 16 cells). Under slip, mismatch lists are gated only by per-depth headroom (no exact-empty requirement) and the global budget is clamped to `480−N`. **90-s from-scratch smoke (hunt, order=model): target 440→best 440, 445→443, 450→444, 455→445, 460→446, 465→436, 470→438 vs baseline 432; sweet spot `--slip-target 460`.** The 446 (e2lib-verified, `eval/best_c5_446_slip.txt`) beats the old all-time from-scratch 433. Mechanism: most restarts die at depth ~4 (~200k restarts/90 s), but survivors carry a ≤20-mismatch prefix to depth ~182 and the hybrid greedy tail closes the rest. Mix-mode 2.5 min: 444 vs 439 baseline.
- **Mix-thread hunt turn is now a node budget, not one restart** (default behavior change, all modes): under slip, restarts die in microseconds, and the old one-restart hunt turn starved hunters against multi-second polish blocks (mix best collapsed to 420/maxdepth 14). A turn now loops `run_restart()` until `node_cap` nodes are spent — identical to old behavior when restarts exhaust the cap.
- **Adaptive restart aborts (`--abort-points "nodes:mindepth,..."`)**: LV's factor-5 heuristic; kills a restart whose `max_depth` lags a node-indexed milestone. With `"2000000:60,10000000:120"` on top of slip-460: same 90-s best (446) at **+50% restarts** (304k vs 200k) — more lottery tickets per hour. Recommended companion to slip.
- **Good groups (`--groups-file`, generator `--mode make-groups`)**: piece-tileability classes per LV — group tileability = #ways 6 distinct group pieces fill a 3×2 interior rectangle (7 internal edges matched; adaptation: our ring is solved separately, so groups are interior-only). `--mode make-groups` hill-climbs 60 groups of 98 (random ~6.5M → ~26M tileability, ~15 min all-cores at `--groups-iters 6000`) → `eval/groups.txt`; precious/useless = top/bottom-16 by cross-group frequency. **LV's hard depth gates are LETHAL here** (his 63/79/96/100 rescaled → avgdepth 7 vs 98, best 427; even useless-forcing alone → avgdepth 63; gates now default off, opt-in via `--group-cks A,B,C,D`). Shipped adaptation: **class-priority bucket ordering** (useless>bad>good>precious, dominant sort key, `--group-order 1` default) — spend worst-tiling pieces early without pruning legal moves. Smoke: neutral (432/94/190 vs 432/98/189 baseline; under slip 444 vs 446, within noise) — harmless, unproven; the honest test is deep-completion frequency in long runs.
- **Column-wise tail scan (`--tail-cols N`)**: LV's optimized search orders end in a column-swept strip. Solo: neutral (433). Under slip: **hurts** (433, maxdepth 142 — tail cells face two decided neighbors exactly where the slip allowance lives, branching dies). Not recommended as-is; LV co-optimized order+slip jointly via a Markov model of per-depth branching (his appendix has the DP pseudocode), which is the missing piece if revisited (Track A logs = the data source).
- **For long runs**: recommended from-scratch/fresh-lineage config = `--slip-target 460 --abort-points "2000000:60,10000000:120"` (scripts unchanged — record hunts resume high seeds where the hunter's job is pool food; slip makes that food 432→446, so likely also right for run5/run1, but only smoke-tested from scratch). LV's remaining unadopted ideas: restart criteria tied to *solution rate* (needs completions to count), order/slip co-optimization model, most-visited-depth abort signal.

**2026-07-08 from-scratch pipeline upgrades** (all in `src/solver.cpp`; several change default behavior):
- **Umbrella bug fixed**: the stagnation resample (`--stagnation-kick`, new flag, default 4000) capped `since_gain` at 4000, so `--umbrella-stagnation 8000` could **never fire** — the umbrella kick had been dead code in every default-flag run. Now lineages at/near the global best (within 1) are exempt from the resample (`Polisher::bump_stagnation`): they keep drifting and `since_gain` keeps counting until the umbrella fires. Band-rebuild trigger changed from `== band_stagnation` to `% band_stagnation == 0` so drifting lineages re-fire it every cycle. Verified: `--stagnation-kick 150 --umbrella-stagnation 450` fires umbrella jumps with no best-score leakage.
- **Hybrid completions** (`--hybrid`, default 1): every hunter restart replays its deepest DFS prefix (captured in `hyb_p/hyb_r` whenever `max_depth` improves) and greedy-fills the tail (`greedy_fill`, refactored out of `greedy_complete`) → every restart emits a fresh-lineage completion instead of nothing. Fixes fresh-pool starvation (previously frozen at the ~8 bootstrap greedy boards, since DFS completions are needle-rare and the completion budget tightens as best rises). Smoke test: from-scratch 421/480 at 15 s, 424 at 90 s (old baseline: 410–414 after 2.5 min; the old 421 plateau took ~4 h). Status: `hyb=`. Composes with `--order model` (deeper stalls → better emitted boards) — **run5/run1.ps1 now pass `--order model --model-file eval\model_weights.txt`**.
- **Pool dedup + plateau archive**: `board_hash` (FNV-1a over cell,piece,rot) dedupes all pools (`SeedPool::have`); new `g_plateau` pool (`--plateau-cap` 4096) archives every distinct board within 1 of the global best (inserted in `bump_stagnation` each step + in `report_completion`) — the 464 author's mass same-score harvest, in-process. Subsumes the old "re-add sideways-drifted equal-score boards" tuning idea. Status: `plat=`.
- **Merge move** (`--merge-pct` 2% of polish steps, `--merge-nodes` 16M): draws two plateau boards, exact-solves their disagreement window on the better one (≤40 cells; outside-window agreement ⇒ identical window piece multisets ⇒ the incumbent-seeded B&B space contains both parents' arrangements) — in-process `plateau_merge.py`. Cross-basin pairs rejected as soon as the diff passes 40 cells; pair dedup via symmetric hash set (`g_merge_tried`). Status: `merge=iters(+gains)`.
- **Tie acceptance** (`--tie-pct`, default **100**, was hardcoded 30): equal-score region/scatter/parity results are now always applied — sideways drift is score-safe (incumbent seeding) and drift speed is the plateau resource. `--tie-pct 30` restores old behavior.
- **RegionSolver dynamic suffix bound**: per-cell top-2 piece values (`cmax1/cmax2/amax_pi`); cells whose argmax piece is already used deeper in the tree contribute only their second-best value. Strictly tighter than the static `suffix_max`, enabling **big-region rolls**: `--region-big-pct` (10) % of region moves use `--region-max-big` (24) cells; `--region-max`/`--region-max-big` now clamped to ≤40 (the array cap).
- Status-line gotcha: `avgdepth` divides by *all* restarts including greedy/bootstrap completions (which contribute depth 0), so mix-mode avgdepth reads roughly half the true DFS stall depth.

**2026-07-06 additions** (all in working tree, defaults preserve old behavior unless noted):
- **Band rebuild** (`BandSolver`, macro-LNS): rebuilds interior rows y0..14 (40–96 cells) via budgeted anytime DFS with the freed pieces, budget = band faults − 1. Whole-board rotations (`rotate_board`, orient 0–3) turn column/top bands into bottom bands; clue cells stay pinned at transformed positions. Fires on polish stagnation (`--band-stagnation 3000`, `--band-nodes 30M`); `--mode band` = systematic sweep for testing; `--selftest` checks rotation identity/score invariance. Inherits the Hunter's move space (every piece matches ≥1 decided W/N neighbor), so its "exhausted = optimal" claims are relative to that space.
- **Umbrella kick** (record author's method: forced leap to distance ~32): on stagnation (`--umbrella-stagnation 8000`), destroy a random 25–40-cell blob, rebuild greedily *without* incumbent seeding, accept unconditionally. Degraded boards never leak to pools/best files (score-gated paths only). `--umbrella 0/1`, `--umbrella-dist 32`.
- **LAHC + exploration**: `--explore-pct 10`% of region moves run non-seeded (`prep_and_solve(..., seed_incumbent=false)` — may return worse); late-acceptance gate `hist[--lahc 50]` accepts/rejects with exact restore. `--lahc 0` disables both.
- **Hungarian assignment bound in RegionSolver**: per-(cell,piece) UB `val[i][p]` (best-rotation fixed-edge matches + internal edges assumed matched) tightens `suffix_max` (was: all decided edges match); root Hungarian relaxation detects tie-only regions (`amax == inc_gain`) → drift-only search at cap/16 (`tieproof=` counter in status). A/B (1 min, noisy): polish iters +41%.
- **Drift snapshots**: `--drift-dir PATH` appends `<score> <edges>` lines (per-thread ring of 64) when a polisher stalls at/near global best — food for plateau merging.
- **`--mode test-polish`**: 60-step single-polisher debug harness with trace (`p.dbg`).

**2026-07-06 raphael-anjou adoptions** (from analyzing https://github.com/raphael-anjou/eternity2, a 219-volume private research program on E2, best 463; changes default behavior where noted):
- **Cost-2 double-break tier** (`--cost2`, default **1**): `Buckets::all_list` + a budget-gated third `try_list(..., true, true)` tier (both W and N mismatch). Motivation: REPLAY experiment proved community strict-460+ boards contain double-break cells one-break-per-cell search can't construct; verified on our boards (JB470 has one at (2,10), R461 at (14,14)) while our old lists (`exact`/`by_w`/`by_n`) could never place one. `1` = BandSolver only (default — band rebuilds around record fault clusters now include the incumbent's family; with it, JB470's orient-3 bands *re-exhaust with no gain* in ≤204k nodes = stronger optimality certificates), `2` = also Hunter DFS (smoke-tested: no throughput loss at 477M nodes/s, but widens tail branching — use when digging near a known double-break family), `0` = off.
- **Parity-class Hungarian move** (`--parity-pct`, default **2**%): `parity_move()` exactly reassigns an *entire chessboard-parity class* of interior non-clue cells (~97 cells — all pairwise non-adjacent so every neighbor is fixed) via the existing O(n³) `hungarian()`; per-entry random cost perturbation (total < one match unit, SCALE=2^14) makes equal-optimum ties into free sideways drift; ties applied at 30% like scatter. Status line: `parity=iters(+gains)`. Coverage no ≤16-cell region or ≤48-cell scatter can reach. Finding: **JB470, Jef_470 and their drift variants are already parity-LAP-optimal in both classes** (after==before always) — another certificate of deep local optimality; the move's value is drift + catching non-optimal parity classes on fresher lineages.
- **`--band-y0-min N`** (default 8 = old behavior): lowers the deepest band-rebuild start row (y0 ∈ [N,12], i.e. up to 14×14=196 freed cells at N=1). Motivation: σ-cycle finding — transitions between record basins are single indecomposable 80–154-cell cycles spanning rows 1–14, structurally out of reach of 42–98-cell bands and 25–40-cell umbrellas. Deep bands make completions needle-rare: pair with a much larger `--band-nodes` and expect an anytime lottery, not exhaustion.

## Exact CP-SAT tools (tools/, need `pip install ortools`)

- `window_solve.py`: fix all cells outside a window (rect / fault-dilation / explicit cells / `--board2` disagreement diff), optimally reassign the window's pieces (rotations free, rim grey-out enforced, clues pinned) via CP-SAT. **`--improve-only` is the workhorse**: pure feasibility ("any strictly better arrangement?") — often decides in *seconds* where maximize mode stays UNKNOWN for 25+ min. Any SAT on a record board = new world record; INFEASIBLE = proof of window-local optimality.
- `window_sweep.py`: resumable sweep of all WxH windows touching ≥1 fault (state file skips decided windows). Windows mixing faults + perfect territory = exhaustive piece-inflow tests. The overnight tool.
- `plateau_merge.py`: pairs same-score drift snapshots (from `--drift-dir`), exact-solves their disagreement sets (typically 4–35 cells) — merges independent plateau discoveries; dedupe state per pair.

**Scarce-color candidate ordering** (`Buckets::build`): candidates in each `exact[w][n]`/`by_w`/`by_n` bucket are stable-sorted (after the randomizing shuffle) by a static per-color rarity weight (`COLOR_RARITY[23]`, computed once in `init_static` from interior-piece color counts), so pieces carrying scarcer colors on their still-open sides are tried before common ones. Came from investigating [Joshua Blackwood's C# solver](https://github.com/jblackwood345/EternityII_Solver) (pure randomized DFS, no LNS — its main tricks are a fixed per-depth break-budget schedule and a 3-color cumulative-count prune). **Important finding**: our own piece set has colors 1–5 appearing only on border/corner pieces (0 interior occurrences, confirmed via `tools/e2lib.py` frequency count) — these are *already* handled exactly by the existing `D[]`/`avail[]` per-color deficit bound in `Hunter`, so porting Blackwood's hard depth-schedule prune directly would have been redundant/unsound. The remaining interior colors (6–22) only vary ~10% in scarcity — too weak to justify a hard cutoff, so a soft value-ordering heuristic was implemented instead of a prune. Smoke-tested (45 s side-by-side, `--clues 5 --mode hunt`): no throughput regression, comparable/slightly better avgdepth; real score impact needs a long user-run to confirm. JV/Hungarian algorithm was also evaluated for the polisher's `scatter_move` LAP — not adopted, since n≤48 there is already sub-millisecond and JV's practical speed edge only matters at n in the hundreds+.

## E2 ground truth

Validated against the e2.bucas.name viewer source in `reference/bucas_init.js`; regenerate data with `tools/make_data.py`, library `tools/e2lib.py`.

- 256 pieces (4 corner / 56 edge / 196 interior), 16×16 board, 22 colors + grey(0). Score = matched internal edges, max 480.
- **Bucas URL encoding**: per-cell 4 letters Up,Right,Down,Left, `'a'+motif`, row-major from top-left. Rotation cw = last char to front: `QUAD[p][k][i] = def[(i-k)&3]`. Format: `https://e2.bucas.name/#puzzle=NAME&board_w=16&board_h=16&board_edges=<1024 letters>&board_pieces=<256×3 digits>&show_conflicts=1`.
- **Clue placements** (0-based row-major cells; pieces 1-based): piece 208 @ cell 34 rot 1; 255 @ 45 rot 3; 139 @ 135 rot 0; 181 @ 210 rot 3; 249 @ 221 rot 1. 1-clue mode = piece 139 @ 135 rot 0 only.
- **Letter maps** to standard letters: jef→std = `atojeqlgbupkfrmhcwvsnid`, jbw→std = `abglqejotchmrfkpudinsvw` (derive with `e2lib.derive_letter_map`).
- Reference boards in `reference/bucas_init.js`: Joshua_Blackwood_470 (honors the center clue → valid 1-clue seed, only clue #3 of 5), Jef_470 (stored under compound key `JBlackwood+Jef_470`), the full 469 family (`JBlackwood+PMcGavin_469`, `_a`, `JBlackwood+Jef_469_a…g`), Joshua_Blackwood_468, Louis_Verhaard_467 (+`_a`,`_b`). - e2lib API gotchas: `score_edges(edges_str)` takes the 1024-char std-letter string (returns `(score, conflicts, border_errors)`); `check_piece_set(data, edges)` returns placements and raises on duplicates; `parse_url(url)` returns the raw params dict (get `params["board_edges"]`).
